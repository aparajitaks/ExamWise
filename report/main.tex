\begin{abstract}
Designing balanced and pedagogically sound examinations is a persistent challenge in education. Educators typically discover that an exam was poorly calibrated only \emph{after} students have already taken it, when it is too late to adjust. This project presents \textbf{ExamWise}, an end-to-end AI-driven educational analytics platform that addresses this problem in two phases. In \textbf{Milestone~1}, we build a machine learning and natural language processing (NLP) pipeline that ingests question--answer data from Stack Overflow, engineers engagement-based difficulty features, and trains a Logistic Regression classifier to predict question difficulty as \textit{Easy}, \textit{Medium}, or \textit{Hard}. The model achieves $65.71\%$ overall accuracy on the test set. In \textbf{Milestone~2}, we extend the system with an \textbf{Agentic AI Assessment Design Assistant} powered by the Google Gemini large language model (LLM). The agent performs stateful analysis of assessment data, retrieves pedagogical best practices through a synthetic Retrieval-Augmented Generation (RAG) mechanism, generates structured six-section assessment improvement reports, and handles incomplete or noisy data gracefully. As an extension, we implement \textbf{Automated Question Generation}, allowing educators to instantly create new exam questions for identified weak topics. The entire application is built using Streamlit and is publicly deployable on free-tier hosting platforms.
\end{abstract}

\section{Introduction}
\label{sec:intro}

\subsection{Motivation and Problem Statement}

In conventional educational assessment workflows, question difficulty is understood only \emph{retrospectively}---after students take the exam and aggregate scores are computed \citep{lord1980applications}. This reactive paradigm creates two significant problems:

\begin{enumerate}[noitemsep]
    \item \textbf{Poorly balanced exams}: Without prior difficulty estimation, exams may be disproportionately easy or hard, failing to differentiate student abilities effectively.
    \item \textbf{Missed learning gaps}: Patterns of student struggle on specific topics are only visible after grading, leaving no opportunity for proactive pedagogical intervention.
\end{enumerate}

\textbf{ExamWise} addresses both problems by building a predictive and generative AI system that:
\begin{itemize}[noitemsep]
    \item Predicts question difficulty from text \emph{before} the exam is administered.
    \item Analyzes difficulty distributions and learning gaps \emph{after} an exam to guide future assessment design.
    \item Generates structured, actionable recommendations grounded in established pedagogical principles.
    \item Automatically creates new questions for topics identified as weak areas.
\end{itemize}

\subsection{Scope of This Report}

This report documents the complete development of the ExamWise system across two milestones:

\begin{itemize}
    \item \textbf{Milestone~1 (Mid-Semester):} ML-based exam question analytics---data pipeline, feature engineering, supervised classification, model evaluation, and basic Streamlit UI.
    \item \textbf{Milestone~2 (End-Semester):} Agentic AI Assessment Design Assistant---LLM integration, synthetic RAG, structured report generation, noisy data handling, and automated question generation.
\end{itemize}

\subsection{Constraints and Requirements}

The project was developed under the following constraints:
\begin{itemize}[noitemsep]
    \item Team size: 3--4 students.
    \item \textbf{No paid APIs permitted}---only free-tier APIs or open-source models.
    \item A user interface is mandatory.
    \item The final application must be publicly hosted (not localhost-only).
\end{itemize}

\section{Literature Review}
\label{sec:litreview}

\subsection{Automated Question Difficulty Estimation}

The estimation of question difficulty from textual and contextual features has been explored through several complementary approaches in educational data mining and psychometrics.

\subsubsection{Item Response Theory (IRT)}
Classical psychometric models, particularly Item Response Theory \citep{hambleton1991fundamentals}, estimate item difficulty as a latent parameter from observed student response patterns. The standard three-parameter logistic (3PL) model defines the probability of a correct response as:

\begin{equation}
    P(\text{correct} \mid \theta) = c + \frac{1 - c}{1 + e^{-a(\theta - b)}}
    \label{eq:irt}
\end{equation}

where $\theta$ is the student ability, $b$ is the item difficulty, $a$ is the discrimination parameter, and $c$ is the guessing parameter. While IRT provides rigorous statistical foundations, it \emph{requires observed response data} and cannot predict difficulty for new, untested items.

\subsubsection{Text-Based Approaches}
Recent work uses NLP features---readability scores, syntactic complexity, word frequency distributions---to predict difficulty from question text alone \citep{ha2019automated}. Common text representations include TF-IDF vectors and pre-trained word embeddings. These approaches enable \emph{a priori} difficulty estimation without requiring student response data.

\subsubsection{Community Q\&A Platforms as Proxies}
Stack Overflow question--answer data has been used as a proxy for educational content difficulty \citep{anderson2012discovering}. Metrics such as answer scores, response rates, and time-to-first-answer serve as engagement-based difficulty indicators. Our approach follows this paradigm, using community engagement to derive difficulty labels.

\subsection{Large Language Models in Education}

The emergence of large language models (LLMs) has created new possibilities for educational technology. LLMs can generate human-quality text, reason about complex topics, and follow structured output formats \citep{brown2020language}. However, they also carry risks of generating plausible but incorrect information (``hallucination''), which is particularly dangerous in educational contexts where accuracy is paramount.

\subsection{Agentic AI Systems}

Agentic AI refers to systems where an AI model operates with explicit state management, goal-directed reasoning, and the ability to use tools or retrieve information \citep{yao2023react}. Unlike simple prompt-response interactions, agentic workflows maintain conversation state across multiple reasoning steps and can incorporate external knowledge bases. Our Assessment Assistant implements this pattern.

\subsection{Retrieval-Augmented Generation (RAG)}

RAG systems augment LLM generation with retrieved context from external knowledge bases, reducing hallucination and grounding responses in verified information \citep{lewis2020rag}. While full RAG systems use vector stores (e.g., FAISS, Chroma) for semantic retrieval, our system implements a \emph{synthetic RAG} approach where curated pedagogical guidelines are injected directly into the prompt context.

\subsection{Data Reduction Strategy}
\label{sec:data-reduction}

Due to memory and computational constraints, the full dataset cannot be processed on standard hardware. We implemented a \textbf{referentially-consistent reduction strategy} in \texttt{scripts/data\_reduction.py} that preserves data integrity:

\begin{algorithm}[H]
\caption{Referentially-Consistent Data Reduction}
\label{alg:reduce}
\begin{algorithmic}[1]
\REQUIRE Raw CSV files, reduction limit $L = 15{,}000$
\STATE Read first $L$ rows from \texttt{Questions.csv}
\STATE Extract question ID set: $\mathcal{Q} = \{q_1, q_2, \ldots, q_L\}$
\STATE \textbf{Stream} \texttt{Answers.csv} in chunks of 50{,}000 rows
\FOR{each chunk}
    \STATE Filter rows where \texttt{ParentId} $\in \mathcal{Q}$
    \STATE Append filtered rows to answer buffer
    \IF{buffer size $\geq L$}
        \STATE \textbf{break}
    \ENDIF
\ENDFOR
\STATE Truncate answer buffer to $L$ rows
\STATE Repeat streaming process for \texttt{Tags.csv}, filtering on \texttt{Id} $\in \mathcal{Q}$
\STATE Save reduced files to \texttt{data/reduced/}
\end{algorithmic}
\end{algorithm}

The chunked reading approach (line 3) ensures that even the full 1.39M-row \texttt{Answers.csv} file can be processed without loading it entirely into memory.

\subsection{Why Stack Overflow Data?}

Stack Overflow questions serve as a reasonable proxy for exam questions because:
\begin{itemize}[noitemsep]
    \item Questions are technical and assessment-like in nature (problem-solving focus).
    \item Community voting (scores) provides a crowd-sourced quality/difficulty signal.
    \item The variety of programming topics mirrors the breadth of a computer science curriculum.
    \item The dataset is freely available, satisfying the no-paid-API constraint.
\end{itemize}

\subsection{Feature Engineering}
\label{sec:feature-eng}

The feature engineering module (\texttt{scripts/feature\_engineering.py}) transforms raw question--answer data into a structured dataset with six computed features per question. The process involves three main steps:

\subsubsection{Step 1: Answer Aggregation}

For each question (identified by \texttt{Id}), all associated answers are grouped by \texttt{ParentId}, and the following statistics are computed:

\begin{table}[H]
\centering
\caption{Engineered features per question}
\label{tab:features}
\begin{tabular}{lll}
\toprule
\textbf{Feature} & \textbf{Computation} & \textbf{Intuition} \\
\midrule
\texttt{max\_answer\_score}   & $\max(\text{answer scores})$       & Quality of the best answer \\
\texttt{avg\_answer\_score}   & $\text{mean}(\text{answer scores})$  & Average answer quality \\
\texttt{answer\_score\_variance} & $\text{var}(\text{answer scores})$ & Disagreement among answers \\
\texttt{answer\_count}        & Count of answers               & Community interest level \\
\texttt{question\_score}      & Score from \texttt{Questions.csv}   & Community rating of the question \\
\texttt{ratio}                & $\frac{\text{max\_answer\_score}}{\text{answer\_count} + 1}$ & Normalised answer quality \\
\bottomrule
\end{tabular}
\end{table}

The aggregation uses \texttt{pandas.groupby()} with named aggregation functions. For questions with a single answer, the variance is set to 0 (since variance is undefined for a single observation).

\subsubsection{Step 2: Difficulty Score Computation}

The core difficulty metric is computed as:

\begin{equation}
    \text{difficulty\_score}_i = 1 - \frac{\text{max\_answer\_score}_i}{\text{max\_answer\_score}_i + \text{answer\_count}_i + 1}
    \label{eq:difficulty}
\end{equation}

\textbf{Intuition:} A question with a high-scoring answer relative to its number of answers is ``easy''---the community quickly converged on a good solution. Conversely, questions with low-scoring answers or very few responses are ``hard''---the community struggled to provide helpful answers.

The score is then normalised to $[0, 1]$ using min--max normalisation:

\begin{equation}
    \text{difficulty\_score}_i^{\text{norm}} = \frac{\text{difficulty\_score}_i - \min(\mathbf{d})}{\max(\mathbf{d}) - \min(\mathbf{d})}
    \label{eq:minmax}
\end{equation}

where $\mathbf{d}$ is the vector of all raw difficulty scores. If all scores are identical (edge case), a fallback value of $0.5$ is assigned.

\subsubsection{Step 3: Difficulty Categorisation}

The normalised scores are mapped to three categorical labels using fixed thresholds:

\begin{equation}
    \text{difficulty\_label}_i =
    \begin{cases}
        \textit{Easy}   & \text{if } \text{difficulty\_score}_i^{\text{norm}} < 0.33 \\
        \textit{Medium} & \text{if } 0.33 \leq \text{difficulty\_score}_i^{\text{norm}} < 0.66 \\
        \textit{Hard}   & \text{if } \text{difficulty\_score}_i^{\text{norm}} \geq 0.66
    \end{cases}
    \label{eq:thresholds}
\end{equation}

\subsection{Text Preprocessing}

Before vectorisation, the question text (\texttt{Title} + \texttt{Body}) undergoes four preprocessing steps:

\begin{enumerate}[noitemsep]
    \item \textbf{HTML tag removal:} Regex pattern \texttt{<[{\textasciicircum}>]*>} strips all HTML markup from the body text.
    \item \textbf{Non-alphabetic character removal:} Retains only alphabetical characters, removing code snippets, numbers, and special characters.
    \item \textbf{Lowercasing:} Converts all text to lowercase for uniform representation.
    \item \textbf{Stop word removal:} English stop words (``the'', ``is'', ``and'', etc.) are excluded during vectorisation.
\end{enumerate}

\subsection{Text Vectorisation}

The cleaned text is transformed into a numerical feature matrix using scikit-learn's \texttt{CountVectorizer} (Bag-of-Words model) \citep{sklearn}. Each question is represented as a sparse vector $\mathbf{x}_i \in \mathbb{R}^{|V|}$ where $|V|$ is the vocabulary size. Each element $x_{ij}$ represents the count of word $j$ in question $i$.

\subsection{Classification Model}

A \textbf{Logistic Regression} classifier is trained for multi-class classification using the softmax (multinomial) formulation:

\begin{equation}
    P(y = k \mid \mathbf{x}) = \frac{e^{\mathbf{w}_k^\top \mathbf{x} + b_k}}{\sum_{j=1}^{K} e^{\mathbf{w}_j^\top \mathbf{x} + b_j}}
    \label{eq:softmax}
\end{equation}

where:
\begin{itemize}[noitemsep]
    \item $\mathbf{x} \in \mathbb{R}^{|V|}$ is the BoW feature vector for a question.
    \item $\mathbf{w}_k \in \mathbb{R}^{|V|}$ is the learned weight vector for class $k$.
    \item $b_k \in \mathbb{R}$ is the bias term for class $k$.
    \item $K = 3$ corresponds to the three classes: Easy, Medium, Hard.
\end{itemize}

The model is trained using L2 regularisation to prevent overfitting on the high-dimensional sparse feature space. The trained model and vectoriser are serialised using \texttt{joblib} \citep{joblib} for deployment.

\subsection{Train--Test Split}

The dataset is split into 70\% training and 30\% testing:
\begin{itemize}[noitemsep]
    \item Training set: 10{,}500 samples
    \item Test set: 4{,}500 samples
\end{itemize}

\subsection{Model Persistence}

Three artifacts are persisted to the \texttt{models/} directory:
\begin{itemize}[noitemsep]
    \item \texttt{model.joblib} (758~KB) --- The trained Logistic Regression classifier.
    \item \texttt{vectorizer.joblib} (447~KB) --- The fitted CountVectorizer with learned vocabulary.
    \item \texttt{metrics.txt} --- The full classification report including per-class precision, recall, and F1.
\end{itemize}

\subsection{Analysis of Results}

\textbf{Key observations:}
\begin{enumerate}
    \item \textbf{Majority class dominance:} The model achieves $65.71\%$ accuracy primarily by correctly classifying \textit{Hard} questions (83\% recall for the class comprising 75.2\% of test data).
    
    \item \textbf{Minority class struggle:} The \textit{Easy} class achieves only 10\% recall and 14\% precision, meaning the model fails to identify most easy questions. This is a direct consequence of having only 473 easy questions in the test set (10.5\%).
    
    \item \textbf{Macro vs.\ weighted metrics:} The macro-averaged F1 of 0.36 (treating all classes equally) is significantly lower than the weighted-averaged F1 of 0.64 (accounting for class prevalence), confirming that performance is driven by the majority class.
    
    \item \textbf{Confusion Matrix interpretation:} The confusion matrix (Figure~\ref{fig:cm}) reveals that most \textit{Easy} and \textit{Medium} questions are misclassified as \textit{Hard}, consistent with the class imbalance.
\end{enumerate}

\section{Methodology: Milestone~2 --- Agentic AI Assistant}
\label{sec:methodology-m2}

\subsection{Objectives}

Milestone~2 extends the ML analytics pipeline into an \textbf{Agentic AI Assessment Design Assistant} that:
\begin{enumerate}[noitemsep]
    \item Analyzes question difficulty distribution and identifies learning gaps.
    \item Retrieves pedagogical and assessment design best practices.
    \item Generates structured assessment improvement recommendations.
    \item Handles incomplete or noisy assessment data gracefully.
    \item Generates new exam questions for weak topics (extension).
\end{enumerate}

\subsection{LLM Selection: Google Gemini}

We selected \textbf{Google Gemini 2.5 Flash} as our LLM backend because:
\begin{itemize}[noitemsep]
    \item It offers a generous \textbf{free-tier API} (satisfying the no-paid-API constraint).
    \item It supports structured output and long context windows.
    \item It provides state-of-the-art reasoning capabilities comparable to GPT-4 class models.
    \item The Python SDK (\texttt{google-generativeai}) is simple to integrate.
\end{itemize}

The \texttt{AssessmentAgent} class (\texttt{scripts/assessment\_agent.py}) implements an agentic workflow with three core components:

\subsubsection{Component 1: Explicit State Management}

The agent maintains explicit state across interactions:

\begin{lstlisting}[language=Python, caption={Agent state initialisation}]
self.state = {
    "is_configured": False,     # API key validated
    "reports_generated": 0,     # Count of reports produced
    "last_identified_gaps": []  # Topics from last analysis

This state management pattern ensures that:
\begin{itemize}[noitemsep]
    \item The agent will not attempt LLM calls without a valid API key.
    \item Usage can be tracked and rate-limited if needed.
    \item Context from previous analyses can inform subsequent interactions.
\end{itemize}

\subsubsection{Component 2: Synthetic Retrieval-Augmented Generation (RAG)}

Rather than using a full vector store (FAISS, Chroma), we implement a \textbf{synthetic RAG} approach where curated pedagogical best practices are stored as a Python dictionary and injected directly into the LLM prompt. This knowledge base contains five established educational principles:

\begin{table}[H]
\centering
\caption{Pedagogical knowledge base entries used for synthetic RAG}
\label{tab:rag-kb}
\begin{tabularx}{\textwidth}{lX}
\toprule
\textbf{Key} & \textbf{Guideline} \\
\midrule
\texttt{difficulty\_distribution} & A standard exam should aim for approximately 20\% easy, 60\% medium, and 20\% hard questions to differentiate student ability while maintaining confidence. \\
\texttt{learning\_gaps} & When students consistently score low on specific topics, it indicates a critical learning gap. Recommend targeted supplementary materials and formative assessment. \\
\texttt{cognitive\_load} & Avoid unnecessary complexity in question phrasing. Clear, concise language ensures we assess the skill, not reading comprehension. \\
\texttt{constructive\_alignment} & Ensure exam questions directly align with the stated learning objectives of the course. \\
\texttt{disclaimer} & This analysis is AI-generated and should serve as an assistive tool, not a definitive authority. \\
\bottomrule
\end{tabularx}
\end{table}

These guidelines are formatted and injected into every prompt, ensuring the LLM's recommendations are \emph{grounded in established pedagogy} rather than generating unsupported claims.

\subsubsection{Component 3: Data Validation and Noisy Data Handling}

Before generating a report, the agent performs four data quality checks on the input DataFrame:

\begin{enumerate}
    \item \textbf{Missing titles:} Counts questions where the \texttt{Title} column is \texttt{NaN}.
    \item \textbf{Missing bodies:} Counts questions where \texttt{question\_body} is \texttt{NaN}.
    \item \textbf{Zero-score anomaly:} If more than 30\% of questions have a difficulty score of exactly 0, this suggests sparse answer data and is flagged.
    \item \textbf{Small sample warning:} If the dataset contains fewer than 100 questions, the agent warns that analysis reliability may be limited.
\end{enumerate}

These quality notes are appended to the data summary and explicitly included in the LLM prompt with the instruction to ``handle gracefully.'' This ensures the generated report acknowledges data limitations rather than presenting potentially misleading conclusions as fact.

\subsection{Structured Report Generation}

The LLM is prompted to generate a report with exactly six sections, matching the project requirements:

\begin{enumerate}
    \item \textbf{Assessment Quality Summary:} Overall evaluation of exam balance.
    \item \textbf{Question Difficulty Distribution:} Statistical summary of Easy/Medium/Hard counts.
    \item \textbf{Identified Learning Gaps:} Topics where students appear to struggle, inferred from the hardest questions.
    \item \textbf{Recommended Assessment Improvements:} Actionable advice for educators.
    \item \textbf{Supporting Pedagogical References:} Citations of the RAG-injected guidelines.
    \item \textbf{Educational and Ethical Disclaimers:} Explicit statement that the analysis is AI-assisted.
\end{enumerate}

\subsection{Prompt Engineering Strategies}

To reduce unsupported educational claims, the following prompt engineering strategies are employed:

\begin{itemize}
    \item \textbf{Role assignment:} The LLM is assigned the role of ``expert Educational Assessment Designer and AI Agent.''
    \item \textbf{Grounding instruction:} ``Avoid making unsupported educational claims outside of the provided guidelines.''
    \item \textbf{Context injection:} All five pedagogical guidelines are included verbatim in every prompt.
    \item \textbf{Structured output format:} The prompt specifies the exact six-section format, preventing the model from generating unstructured or tangential content.
\end{itemize}

\subsection{Extension: Automated Question Generation}

As our chosen extension feature, the system can generate new exam questions on demand. Given a topic name (e.g., ``Dynamic Programming'') and a difficulty level (Easy/Medium/Hard), the agent prompts Gemini to produce:

\begin{enumerate}[noitemsep]
    \item A question title.
    \item A detailed problem description with constraints.
    \item Sample input/output examples.
    \item A pedagogical justification explaining why the question effectively tests the concept.
\end{enumerate}

This feature closes the feedback loop: the Assessment Report identifies weak topics $\rightarrow$ the educator uses Automated Question Generation to create targeted questions $\rightarrow$ future exams are better balanced.

\section{Discussion}
\label{sec:discussion}

\subsection{Strengths}

\begin{enumerate}
    \item \textbf{End-to-end integration:} The system seamlessly connects data ingestion, ML prediction, AI-powered analysis, and interactive UI in a single application.
    \item \textbf{Modular architecture:} Each pipeline stage (reduction, engineering, prediction, analysis) is encapsulated in a separate module, enabling independent testing and future improvements.
    \item \textbf{Pedagogical grounding:} The synthetic RAG approach ensures that AI recommendations are anchored in established educational principles, not hallucinated claims.
    \item \textbf{Graceful degradation:} The system handles missing data, empty datasets, and absent API keys without crashing, providing informative warnings instead.
    \item \textbf{Free-tier compliance:} The entire system runs on free-tier services (Gemini Flash, Streamlit Cloud), making it accessible and cost-effective.
\end{enumerate}

\subsection{Limitations}

\begin{enumerate}
    \item \textbf{Class imbalance (Milestone~1):} The 75\% Hard class dominance causes the classifier to be heavily biased. Techniques such as SMOTE oversampling, class weight adjustment (\texttt{class\_weight='balanced'}), or stratified sampling should be explored.
    
    \item \textbf{Single model baseline:} Only Logistic Regression was trained. Comparing with Decision Trees, Random Forests, SVMs, or gradient-boosted models would provide more robust baselines and potentially better minority-class performance.
    
    \item \textbf{Simple vectorisation:} CountVectorizer captures word presence but ignores word importance. TF-IDF weighting or contextual embeddings (BERT) would improve discriminative power.
    
    \item \textbf{Proxy difficulty labels:} Stack Overflow engagement metrics are an imperfect proxy for actual exam difficulty. Real student response data would provide more accurate ground-truth labels.
    
    \item \textbf{LLM dependency:} The Milestone~2 agent requires an active internet connection and a valid API key. Offline operation is not supported.
    
    \item \textbf{Synthetic vs.\ full RAG:} The hardcoded knowledge base is limited to five guidelines. A full vector-store RAG system with a larger pedagogical corpus would improve the depth and specificity of recommendations.
\end{enumerate}

\section{Future Work}
\label{sec:future}

\begin{enumerate}
    \item \textbf{Advanced NLP models:} Replace CountVectorizer with TF-IDF or pre-trained transformer models (e.g., BERT, DistilBERT) for richer text representations that capture semantic meaning.
    
    \item \textbf{Class imbalance handling:} Apply SMOTE, ADASYN, or adjust \texttt{class\_weight} in scikit-learn to improve minority class recall.
    
    \item \textbf{Multi-model comparison:} Train and evaluate Decision Trees, Random Forests, SVM, and XGBoost classifiers to identify the best-performing model.
    
    \item \textbf{Full RAG pipeline:} Integrate a vector store (Chroma or FAISS) with a comprehensive pedagogical knowledge base sourced from education research papers.
    
    \item \textbf{LangGraph agent:} Migrate the agentic workflow to LangGraph for more sophisticated multi-step reasoning with tool use capabilities.
    
    \item \textbf{Real exam data:} Partner with educational institutions to collect actual student response data for ground-truth difficulty calibration.
    
    \item \textbf{PDF export:} Implement PDF generation of the assessment report for formal documentation.
    
    \item \textbf{Multi-exam comparative analysis:} Enable side-by-side comparison of multiple exam datasets to track assessment quality trends over time.
\end{enumerate}

\section{Conclusion}
\label{sec:conclusion}

This project demonstrates that combining classical machine learning with modern generative AI creates a powerful, practical tool for educational assessment design.

\textbf{Milestone~1} established a complete data pipeline from raw Stack Overflow data to difficulty predictions, achieving $65.71\%$ accuracy with a Logistic Regression baseline. While limited by class imbalance and simple vectorisation, it provides a functional foundation and clearly identifies the challenges inherent in automated difficulty estimation.

\textbf{Milestone~2} transformed these quantitative predictions into qualitative, actionable insights using a stateful Gemini-powered agent. By injecting pedagogical guidelines via synthetic RAG and implementing robust noisy data handling, the system produces trustworthy, grounded recommendations that respect the boundaries of AI-assisted analysis. The Automated Question Generation extension closes the assessment improvement loop, enabling educators to immediately create targeted content for identified weak areas.

Together, the two milestones deliver a complete assessment lifecycle tool---from retrospective analysis of past exams to proactive design of future ones---accessible through a clean, interactive Streamlit interface and deployable on free-tier cloud platforms.

# difficulty_score = 1 - (max_answer_score / (max_answer_score + answer_count + 1))
df['difficulty_score'] = 1 - (
    df['max_answer_score'] / 
    (df['max_answer_score'] + df['answer_count'] + 1)
)

# Normalize between 0 and 1
ds_min = df['difficulty_score'].min()
ds_max = df['difficulty_score'].max()
if ds_max > ds_min:
    df['difficulty_score'] = (df['difficulty_score'] - ds_min) / (ds_max - ds_min)
else:
    df['difficulty_score'] = 0.5  # Fallback

# Categorize
def categorize(score):
    if score < 0.33: return 'easy'
    elif score < 0.66: return 'medium'
    return 'hard'

df['difficulty_label'] = df['difficulty_score'].apply(categorize)
\end{lstlisting}
